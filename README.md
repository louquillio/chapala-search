# Chapala Forum Search

Client-side vector search engine for the [Ajijic/Chapala/Guadalajara forum](https://chapala.com/webboard/index.php?/forum/1-ajijicchapalaguadalajara/) on chapala.com/webboard.

## Architecture

```
┌──────────────────────────────────────────────────────┐
│              Local Workstation (cron)                 │
│                                                      │
│  scraper/build-index.js                              │
│    ├─ Crawls forum pages                             │
│    ├─ Parses qualified threads (≥3 reactions)        │
│    ├─ Chunks text by paragraphs                      │
│    └─ Generates 384-dim embeddings via all-MiniLM    │
│       (local ONNX — no API calls needed)             │
│          │                                           │
│          ▼                                           │
│  public/index.json  ─── Vite build ───► docs/        │
│                                              │       │
│         GitHub Pages ◄── git push ◄──────────┘       │
│              │                                       │
└──────────────┼───────────────────────────────────────┘
               ▼
┌──────────────────────────────────────────────────────┐
│              User's Browser (client-side)             │
│                                                      │
│  - Loads index.json + ONNX model                     │
│  - Runs transformers pipeline in-browser             │
│  - Computes cosine similarity locally                │
│  - Zero backend, zero API calls                      │
└──────────────────────────────────────────────────────┘
```

This is a **zero-backend** architecture:
- **Scraping & indexing** runs locally on your workstation (weekly cron)
- **Storage** is a static JSON file with pre-computed vectors
- **Search** runs entirely in the browser — the user types a question, the browser computes the query embedding and finds the closest matches via cosine similarity

## Requirements

- **Node.js** 18+ (tested on v26)
- **npm** (ships with Node)
- **Git** (for deployment)
- A **GitHub** account and repository for GitHub Pages hosting (optional — works locally too)

No API keys needed. Embeddings are generated locally using [all-MiniLM-L6-v2 ONNX](https://huggingface.co/onnx-community/all-MiniLM-L6-v2-ONNX) via the `@huggingface/transformers` library.

## Local Development

### 1. Install dependencies

```bash
npm install
```

### 2. Run a historical crawl

Scrape the first 100 pages of the forum to build a comprehensive index:

```bash
node scraper/build-index.js --pages=100
```

This will take a while (the model downloads on first run — ~45 MB). Subsequent runs are faster since the model is cached.

### 3. Start the dev server

```bash
npm run dev
```

Opens at `http://localhost:5173` with live reload.

### 4. Build for production

```bash
npm run build
```

Output goes to `docs/`. Open `docs/index.html` directly or serve with:

```bash
npm run preview
```

## Automation

### Environment Variables

None required for the core pipeline. The `HEALTHCHECKS_URL` variable is optional — set it if you use [Healthchecks.io](https://healthchecks.io) for dead-man's switch monitoring:

```bash
export HEALTHCHECKS_URL="https://hc-ping.com/your-uuid-here"
```

### Cron Setup

To run the pipeline every Sunday at midnight:

```bash
crontab -e
```

Add the line:

```
0 0 * * 0  /home/your-user/projects/chapala-search/scraper/run-pipeline.sh
```

The script handles:
1. Locking (prevents concurrent runs)
2. Scraping the latest 3 pages of forum topics
3. Building the static site
4. Committing and pushing to GitHub if new data was indexed
5. Pinging a Healthchecks.io endpoint on success

### Manual Pipeline Run

```bash
./scraper/run-pipeline.sh
```

Check the log at `scraper/pipeline.log` for output.

## How the Indexing Works

1. **Crawl** — The scraper navigates the forum's topic listing pages (up to `--pages=N`), collecting every non-pinned thread URL
2. **Fetch** — Each thread page is loaded and parsed
3. **Score** — Every post in each thread is checked for reactions. Only posts with ≥3 total reactions (Likes + Thanks — called "points") are kept
4. **Extract** — Qualifying posts have their text extracted, HTML stripped
5. **Chunk** — Text is split by double newlines; paragraphs over 400 chars are broken at sentence boundaries
6. **Embed** — Each chunk is converted to a 384-dimensional vector using all-MiniLM-L6-v2 (ONNX, local)
7. **Index** — Each chunk becomes an entry in `public/index.json` with its text, thread title, thread URL, direct post anchor (`#findComment-XXXXX`), post date, points, and vector

## How Search Works (in the Browser)

1. **Load** — The page fetches `index.json` and loads the `all-MiniLM-L6-v2` ONNX model via `@huggingface/transformers`
2. **Encode** — When the user types a query, it's embedded into a 384-dim vector using the same model
3. **Score** — Every entry in the index is scored using four signals:

   ```
   score = match% × 0.55  +  points_norm × 0.25  +  recency × 0.20  +  term_bonus (+0.15)
   ```

   | Signal | Weight | Description |
   |---|---|---|
   | **Match %** | 55% | Cosine similarity between query vector and entry vector. Higher = closer in meaning |
   | **Points** | 25% | Community reactions (Likes + Thanks), normalized against the max in the dataset |
   | **Recency** | 20% | Linear decay over 2 years — a post from today scores 1.0, from 1 year ago scores 0.5 |
   | **Term bonus** | +0.15 | If the query terms appear literally in the post text, this post gets a flat boost |

4. **Filter** — Two-tier threshold:

   - Posts containing any query term → shown if match > 10%
   - Posts without query terms → shown only if match > 35%

   This prevents noise: only posts that are genuinely semantically close (or literally match) will appear.

5. **Group** — Results are grouped by thread. The best-scoring post from each thread is shown prominently; additional posts from the same thread are collapsed behind a toggle
6. **Link** — Each result links directly to the specific post via `#findComment-XXXXX`, not to the top of the thread

Everything runs in your browser. No data is sent to any server — your query never leaves your device.

## GitHub Pages Deployment

This repo is configured to serve from the `docs/` folder on the `main` branch:

1. Push to GitHub
2. Go to repo Settings → Pages
3. Source: **Deploy from a branch**
4. Branch: `main`, folder: `/docs`
5. Site will be live at `https://<user>.github.io/chapala-search/`

The `run-pipeline.sh` script auto-commits and pushes when new data is indexed, so the site updates automatically after each weekly crawl.

> **Note:** The remote is configured for SSH (`git@github.com:louquillio/chapala-search.git`). Your SSH key is already set up, so the pipeline's auto-push works headlessly in cron with no extra credential step.
